Python Choices Explanation Report
Project: MCP CNN project
Date: 2026-03-05

=================================================================
1) Purpose of this report
=================================================================

This report explains why specific Python choices were made in the project code.
It covers language-level choices, library choices, architectural choices, and runtime behavior choices across the `src/` modules.


=================================================================
2) Global Python choices used across files
=================================================================

2.1 `from __future__ import annotations`
- Where: all main source files.
- Choice: postponed evaluation of type annotations.
- Why:
  - avoids runtime issues with forward references
  - keeps typing clean and lightweight at runtime
  - good compatibility pattern for static analysis tools

2.2 Type hints (`dict[str, Any]`, `tuple[int, int]`, etc.)
- Choice: explicit typing for function signatures.
- Why:
  - improves readability and intent clarity
  - helps IDE autocomplete and linting
  - lowers risk of misuse of config/data structures

2.3 `pathlib.Path` over string path concatenation
- Choice: use `Path` for filesystem operations.
- Why:
  - safer and clearer path manipulation
  - OS-agnostic behavior
  - easier parent/exists checks and file IO operations

2.4 `argparse` for CLI interfaces
- Choice: each runnable script defines `parse_args()`.
- Why:
  - standard library dependency (no extra package)
  - predictable interface for automation
  - explicit contract for config/image arguments

2.5 Centralized config loading
- Choice: `src/utils/config.py::load_yaml` used by all entrypoints.
- Why:
  - single source of truth for config parsing
  - shared error behavior (`FileNotFoundError`, schema-type guard)
  - easier future enhancements (schema validation)

2.6 Centralized logging utility
- Choice: `src/utils/logging_utils.py::get_logger`.
- Why:
  - consistent log formatting across modules
  - avoids duplicated logger setup code
  - practical for monitoring train/eval runs


=================================================================
3) Library choices and why they fit this project
=================================================================

3.1 TensorFlow / Keras
- Choice: TensorFlow for model, training, evaluation, and tf.data pipelines.
- Why:
  - integrated end-to-end DL workflow
  - easy model save/load and callback support
  - mature image pipeline tools and deployment options

3.2 NumPy
- Choice: used for inference argmax/probability handling.
- Why:
  - lightweight numerical array operations
  - standard for model post-processing

3.3 Pandas
- Choice: export training history to CSV.
- Why:
  - simple conversion from `history.history` dict
  - straightforward downstream plotting/reporting

3.4 PyYAML
- Choice: parse YAML config files.
- Why:
  - human-readable experiment config format
  - cleaner than hardcoded constants in code

3.5 PIL (Pillow)
- Choice: single-image loading/resizing for inference and GUI preview.
- Why:
  - robust image format support
  - simple API for preprocessing and UI thumbnails

3.6 Tkinter
- Choice: desktop GUI in `gui_predict.py`.
- Why:
  - built-in with Python (no external web framework needed)
  - enough for a local uploader + prediction interface
  - fast to integrate for assignment/demo workflows


=================================================================
4) File-by-file Python choices
=================================================================

4.1 `src/utils/config.py`

Choice A: `yaml.safe_load`
- Why: safer than full loader for untrusted YAML content.

Choice B: strict dict check after loading
- Why: protects downstream code expecting mapping access.

Choice C: explicit file existence check
- Why: fail-fast with actionable error.


4.2 `src/utils/logging_utils.py`

Choice A: named logger factory
- Why: module-specific logs while sharing one format.

Choice B: handler guard (`if logger.handlers`)
- Why: prevents duplicate handlers when module is imported repeatedly.

Choice C: custom formatter with timestamp + level + module
- Why: easier debugging for long-running training/evaluation.


4.3 `src/features/preprocess.py`

Choice A: convert image to RGB explicitly
- Why: normalizes channel assumptions for model input.

Choice B: normalize to [0,1] with float32
- Why: matches training pipeline normalization.

Choice C: add batch dimension (`expand_dims`)
- Why: Keras model expects shape `[batch, H, W, C]`.


4.4 `src/data/dataset.py`

Choice A: custom file indexing helper (`_list_images_and_labels`)
- Why:
  - full control over class discovery and file filtering
  - avoids runtime issues previously seen with legacy loader pools

Choice B: private helper naming (`_make_dataset`)
- Why: internal implementation detail, not public API.

Choice C: tf.data pipeline from tensor slices
- Why:
  - explicit and scalable data pipeline
  - good performance with map/batch/prefetch
  - avoids semaphore/threadpool limitations encountered in this environment

Choice D: use `AUTOTUNE` for map/prefetch
- Why: delegates pipeline optimization to TensorFlow runtime.

Choice E: one-hot encoding labels in loader
- Why: directly compatible with categorical cross-entropy.

Choice F: class consistency check between train/validation
- Why: avoids silent train/val label misalignment.

Choice G: `infer_class_names()` helper
- Why: keeps inference scripts independent of hardcoded class labels.


4.5 `src/models/model.py`

Choice A: dedicated `build_cnn` function
- Why:
  - architecture isolated from training logic
  - easier swapping/testing of model variants

Choice B: configurable model hyperparameters via `cfg`
- Why: architecture is controlled by YAML, not code edits.

Choice C: Sequential API
- Why: model topology is linear and easy to read for educational baseline.

Choice D: Dropout layer
- Why: regularization against overfitting.


4.6 `src/models/train.py`

Choice A: split `parse_args()` and `main()`
- Why: cleaner testing/readability and standard CLI structure.

Choice B: deterministic seed setup (`tf.random.set_seed`)
- Why: improves reproducibility.

Choice C: callback composition
- EarlyStopping with `restore_best_weights=True`
  Why: final in-memory model reflects best val-loss epoch.
- ModelCheckpoint (`save_best_only=True`)
  Why: persist best validation-accuracy checkpoint.

Choice D: export history as CSV
- Why: keeps an analyzable training record independent of notebook state.

Choice E: optional secondary model save (`secondary_model_path`)
- Why:
  - supports dual formats (`.h5` + `.keras`)
  - improves compatibility across tools/runtime preferences


4.7 `src/models/evaluate.py`

Choice A: isolated evaluation entrypoint
- Why: clean separation of training and benchmarking concerns.

Choice B: metrics dict JSON serialization
- Why: standardized machine-readable report for automation.

Choice C: Path parent creation before write
- Why: robust file output even on fresh environments.


4.8 `src/models/predict.py`

Choice A: class inference from train directory
- Why: avoids hardcoding labels.

Choice B: prints full probability distribution
- Why: gives interpretability beyond top-1 class.

Choice C: keeps CLI output simple text
- Why: easy integration into shell scripts and demos.


4.9 `src/models/gui_predict.py`

Choice A: GUI entrypoint with `main()`
- Why: consistent structure with other modules.

Choice B: path resolution helpers (`resolve_config_path`, `resolve_project_path`)
- Why:
  - robust execution from project root or subdirectories
  - avoids common relative-path failures

Choice C: `sys.path` bootstrap for direct script mode
- Why: allows `python gui_predict.py` without package import errors.

Choice D: callback-based upload/predict flow
- Why: idiomatic event-driven Tkinter design.

Choice E: thumbnail preview via PIL
- Why: UI usability and instant visual feedback.

Choice F: show predicted class + confidence (labeled as Accuracy)
- Why: clear user-facing result format for quick interpretation.

Choice G: explicit buttons (`Upload New Image`, `Quit`)
- Why: direct and user-friendly interaction model.


=================================================================
5) Why these Python choices are coherent together
=================================================================

These choices combine into a coherent engineering style:
- Standard library where possible (`argparse`, `pathlib`, `tkinter`)
- Focused third-party dependencies only where needed (TF, NumPy, Pandas, PIL, YAML)
- Modular code boundaries with low coupling
- Reproducible config-driven runtime
- Fail-fast behavior for missing resources
- Practical outputs for reporting and automation


=================================================================
6) Tradeoffs of current Python choices
=================================================================

6.1 Class labels inferred from folder names
- Benefit: no hardcoded classes.
- Tradeoff: can drift if folders are modified after training.

6.2 Tkinter GUI
- Benefit: zero web stack, quick local use.
- Tradeoff: less portable for remote/browser-based deployment.

6.3 Sequential CNN baseline
- Benefit: clarity and simplicity.
- Tradeoff: may underperform transfer-learning backbones.

6.4 YAML without formal schema validation library
- Benefit: lightweight.
- Tradeoff: malformed key structures may fail later in runtime.


=================================================================
7) Suggested Python-level improvements
=================================================================

1. Add dataclass-based typed config wrappers
- Improves key safety and editor guidance.

2. Persist class mapping JSON during training
- Prevents label-order mismatch in future inference.

3. Introduce shared inference service function
- Reuse same prediction core between CLI and GUI to reduce duplication.

4. Add richer exception classes for config/path errors
- Improves user feedback and test granularity.

5. Add module-level docstrings for all public modules
- Better maintainability and generated documentation quality.


=================================================================
8) Conclusion
=================================================================

The current Python choices are appropriate for a structured ML project aimed at both academic clarity and practical usability. The code uses strong defaults: modular boundaries, clear runtime entrypoints, consistent config handling, and explicit data/model flow. The architecture is extensible and can be upgraded incrementally without major rewrites.

End of report.
